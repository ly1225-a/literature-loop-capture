# Literature Loop Capture

Run a review as an auditable loop: plan queries, review them with the user,
search publisher result pages through OpenCLI, screen titles/abstracts, capture
full text only after approval, write reading notes, score coverage, iterate
when gaps remain, then export a LLM Wiki project.

## Preconditions

- Use a project-local Python environment and install the package requirements.
- Run `opencli doctor`, select the intended Chrome profile, and log into the
  needed publisher sites in that profile before discovery or capture.
- Set `OPENALEX_API_KEY` for normal query-plan grounding. Keep keys in the
  shell or a local secret manager, never in tracked files.
- Supported structured publisher routes are direct URLs for
  ScienceDirect/Elsevier, ACS, Wiley, and Springer. Other sources become
  `manual_hold` unless the user explicitly expands the route set.

## Primary Commands

```bash
.venv/bin/python literature-loop-capture/scripts/literature_loop.py doctor
.venv/bin/python literature-loop-capture/scripts/literature_loop.py plan "Review <topic>, years <start>-<end>, publishers Elsevier, ACS, Wiley, Springer." --rounds 3 --year-start <start> --year-end <end>
.venv/bin/python literature-loop-capture/scripts/literature_loop.py review-plan "LiteratureCaptures/<run-folder>"
.venv/bin/python literature-loop-capture/scripts/literature_loop.py resume "LiteratureCaptures/<run-folder>"
.venv/bin/python literature-loop-capture/scripts/literature_loop.py verify "LiteratureCaptures/<run-folder>"
```

After `review-plan`, serve the run directory and open
`query-plan-review.html` in the Codex built-in browser, not the OpenCLI Chrome
profile:

```bash
python3 -m http.server 8765 --bind 127.0.0.1 --directory "LiteratureCaptures/<run-folder>"
```

Open `http://127.0.0.1:8765/query-plan-review.html`. Record user comments with
`review-plan --target ... --severity correction --note ...`; resolve them
before approval.

## Loop Contract

Use one responsible agent per atomic subquestion. A subquestion is terminal
only when its coverage decision is `sufficient`, `stop_with_gaps`, or
`blocked`.

Normal loop:

```text
research question
-> English normalization
-> OpenAlex metadata grounding
-> agent-authored query plan
-> HTML query-plan review and approval
-> OpenCLI publisher discovery
-> title/snippet triage
-> abstract preview
-> approved full-text capture
-> reading-note-zh.md
-> seed/reference/gap extraction
-> coverage review
-> sufficient, stop_with_gaps, blocked, or approved next broad query
```

Title/snippet/abstract evidence is screening evidence only. Coverage can be
decided only from captured full text, reading notes, seed ledgers, references,
gap review, and coverage artifacts.

## Exact Targets

Exact targets come from reading notes, high-value seeds, selected references,
coverage gaps, query rationale, and user corrections. Python must not turn a
short or polysemous label into an OpenAlex capture target by itself.

Before any exact target enters a capture queue, the agent must record:

- original seed text
- agent-verified exact query
- entity type
- expected title terms
- required context terms
- negative senses for ambiguous names
- selected OpenAlex candidate evidence
- `agent_openalex_verified=true`

Use this rule for short database/resource names where the same string can also
mean an unrelated ordinary term.

## References, PDFs, And MinerU

Reference follow-up starts after primary subquestions are terminal. Python may
dedupe, route, and build queues, but the reading-note recommendation is the
intellectual approval.

For unsupported references or missing publisher full text:

1. Build or refresh `_knowledge/`.
2. Put user-supplied PDFs in the relevant
   `_knowledge/subquestions/<id>/manual_pdf_dropbox/`.
3. Sync PDFs into canonical article folders.
4. Normalize with MinerU API.
5. Continue reading notes and coverage review from normalized Markdown.

Do not copy PDFs into the LLM Wiki export when normalized Markdown already
exists.

## LLM Wiki Export

After `overview.md`, subquestion summaries, reading notes, and coverage gates
are complete, run:

```bash
.venv/bin/python literature-loop-capture/scripts/llm_wiki_export.py "LiteratureCaptures/<run-folder>" --output-root "<export-root>"
```

The export keeps `wiki/` lightweight: overview, index, log, subquestions,
queries, and ledgers. Article-level source pages are generated by LLM Wiki
ingest from `raw/sources/articles/**/article.md`; figures and tables live under
`raw/assets/`.
